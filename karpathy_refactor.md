# Centaur Brain: Karpathy Refactor Implementation Plan

This document outlines the detailed software implementation plan to evolve Centaur Brain from an automated ingestion pipeline into a persistent, compounding knowledge graph, inspired by Andrej Karpathy's "LLM Wiki" concept.

## Phase 1: The "Entity Update" Step (Ontology Evolution)
**Goal:** When a new source is ingested, automatically update existing concept pages (`[[Concept]]`) instead of just linking to them. This creates the compounding knowledge effect.

### 1.1 Backend Architecture Changes (`backend/main.py`)
- **Current Flow:** API receives payload -> Saves raw -> Prompts Gemini for summary -> Writes `{Title}.md`.
- **New Flow:**
  1. API receives payload -> Saves raw.
  2. Prompts Gemini for summary **AND** requests a structured JSON list of Core Concepts.
  3. Write the `{Title}.md` summary to `Vault/02 Summaries/`.
  4. **Entity Processing Loop:** For each concept in the extracted JSON list:
     - Sanitize the concept name for a valid filename.
     - Check if `Vault/04 Concepts/{Concept}.md` exists.
     - **If it does NOT exist:** Create a new page `Vault/04 Concepts/{Concept}.md` with a baseline definition generated by Gemini, linking back to the `{Title}.md` source.
     - **If it DOES exist:** Read the existing `Vault/04 Concepts/{Concept}.md`. Send its contents to Gemini along with the new `{Title}.md` summary.
     - *Prompt:* "Here is the existing concept page for '{Concept}'. Here is a new source summary that references it. Please update the concept page to integrate any new information, note contradictions, and add a backlink to the new source. Return the updated Markdown."
     - Overwrite `Vault/04 Concepts/{Concept}.md` with the updated content.

### 1.2 Prompt Engineering (Detailed Summaries & Flexible Concept Extraction)
- Modify the current "Ontology Architect" prompt to use `response_mime_type="application/json"` or structured outputs.
- **Enhance Detail:** Explicitly instruct the LLM to write "comprehensive, highly detailed summaries that preserve nuance and specific arguments, rather than over-simplified high-level overviews. Err on the side of providing more detail."
- **Flexible Concept Extraction:** Instruct it to "extract all non-trivial concepts, themes, and entities. The number of concepts extracted should scale organically with the length and conceptual density of the source material. Extract fewer concepts for simple articles and many concepts for rich, complex sources, ensuring no valuable idea is left behind."
- Schema should return:
  - `summary_markdown`: The formatted markdown for the summary page.
  - `concepts`: An array of strings representing the `[[wikilinks]]` to process.

### 1.3 Architecture Note: Atlas Themes vs. Concepts vs. Tags
- **The Obsidian Graph Structure:** To maintain a visually distinct and hierarchical knowledge graph, we will use three explicit folders. In Obsidian's Graph View, you will use Group colors based on these paths to recreate your existing aesthetic (e.g., `path:"03 Atlas"` = Red, `path:"04 Concepts"` = Green, `path:"02 Summaries"` = Grey).
- **Atlas Themes (`03 Atlas/`):** These are your high-level, recurring macro-categories (e.g., `[[Artificial Intelligence]]`, `[[Leadership]]`).
  - *Dynamic Discovery:* During ingestion, the backend will dynamically read the filenames in the `03 Atlas/` directory and pass them to Gemini as your "Established Themes."
  - *Adding Themes:* If you want a new Theme (e.g., "Software"), you simply create an empty `Software.md` file in `03 Atlas/`. The AI will automatically start linking new summaries and concepts to it.
- **Concepts (`04 Concepts/`):** These are the granular ideas (formerly your topic `#tags`) extracted from sources. Because they are now full Markdown pages, they can contain relationships.
  - *The Hierarchy:* When generating a summary, Gemini will link the `02 Summary` to specific `04 Concepts`, and it will ensure that the `04 Concepts` link *up* to the relevant `03 Atlas` Themes.
- **Tags (`#tags`):** We **still need tags**, but they are strictly demoted to **metadata and broad filtering**, not ontology or topics. Use tags for states and types (e.g., `#type/book`, `#type/article`, `#status/processed`). They should generally be hidden from the main Graph View to reduce noise.

## Phase 2: The "Lint" / "Janitor" Pass (Knowledge Base Maintenance)
**Goal:** Periodically scan the Vault to fix broken links, create missing pages, and flag contradictions.

### 2.1 Expand `backend/janitor_clean.py`
- **Orphan Link Detection (The Indexer):**
  - Parse all markdown files in the Vault using Regex to find all `[[links]]`.
  - Compare the list of found links against actual `.md` files in the `04 Concepts/` folder.
  - Generate a list of "Missing Concepts" (links that have no corresponding file).
- **Auto-Healing (AI-Driven):**
  - For the top N most frequently linked missing concepts, prompt Gemini: "Generate a concept page for '{Concept}' based on its usage in the following context snippets: [snippets extracted from the vault]."
  - Automatically create these files in `Vault/04 Concepts/`.
- **Contradiction/Synthesis Scan:**
  - Pick concept pages that have grown beyond a certain file size (e.g., > 500 words).
  - Prompt Gemini to "Refactor and organize this concept page, checking for internal contradictions and improving structure."

## Phase 3: Auto-Updating `log.md` (Chronological Timeline)
**Goal:** Maintain an append-only timeline of all AI actions for observability.

### 3.1 Logging Utility (`backend/logger.py`)
- Create a simple utility module to handle appending to `Vault/log.md`.
- **Format:** `## [{YYYY-MM-DD HH:MM}] {action_type} | {details} | Concepts: {concepts}`
- **Example:** `## [2026-04-06 14:30] Ingested | Source: "The Bitter Lesson" | Concepts: [[AI]], [[Compute]]`
- **Integration Points:**
  - Call this in `main.py` after a successful ingestion.
  - Call this in `janitor_clean.py` after a linting pass (e.g., `## [2026-04-06 02:00] Lint | Created 5 missing concept pages`).

## Phase 4: Conversational Querying via Gemini CLI
**Goal:** Allow users to have an ongoing, interactive chat with their vault and save valuable answers as permanent nodes.

### 4.1 `GEMINI.md` Integration
Instead of building a separate conversational CLI tool from scratch, we will leverage the existing Gemini CLI.
- **Setup:** Create a `GEMINI.md` file in the root of the Obsidian Vault (or the Centaur Brain project).
- **Persona & Instructions:** The `GEMINI.md` will instruct the Gemini CLI to act as the "Centaur Brain Knowledge Navigator".
- **Workflow:**
  1. The user opens a terminal in the Vault and runs `gemini` to start an interactive chat.
  2. The `GEMINI.md` instructions will tell the CLI agent to aggressively use its tools (like `grep_search` and `read_file`) to search the `Vault/` for context when the user asks a question.
  3. The agent will synthesize answers based on the local Markdown files.
  4. **The "Query to Artifact" Loop:** The instructions will explicitly tell the agent: "After providing a synthesized answer, ask the user if they would like to save this synthesis to the Vault. If they say yes, use your tools to save it as a new Markdown file in `Vault/02 Summaries/` (or a dedicated folder) and append a record to `log.md`."

## Phase 5: Recommended Implementation Order
1. **Phase 3 (Logging):** Easiest to implement and provides immediate observability for all future phases.
2. **Phase 1 (Entity Update):** The core "compound knowledge" feature. Requires careful prompt engineering and JSON parsing but delivers the most value.
3. **Phase 4 (Conversational Querying):** High value, minimal-code addition. Just requires writing the `GEMINI.md` prompt and utilizing the Gemini CLI.
4. **Phase 2 (Janitor):** Can be built iteratively. Start with orphan link detection before moving to AI auto-healing.
